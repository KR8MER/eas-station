About EAS Station

Software-defined drop-in replacement for commercial EAS encoder/decoders built on Raspberry Pi-class hardware

Development-Only Notice: EAS Station is actively pursuing feature parity with commercial encoder/decoders but remains experimental. It is not yet FCC-certified, is not ready for production deployment, and must only be evaluated in controlled lab and training environments until certification paths are explored.

EAS Station is a comprehensive, open-source emergency alert processing system designed for broadcasters, emergency communications professionals, and public safety organizations. It integrates CAP (Common Alerting Protocol) feeds, SAME (Specific Area Message Encoding) audio generation, and multi-channel alert distribution to provide real-time emergency notifications—all with the explicit goal of replacing utilitarian hardware appliances with a disciplined software stack running on Raspberry Pi-class platforms, SDR receivers, and a handful of HATs.

Mission & Purpose

EAS Station was created to bridge the gap between modern digital alert systems and traditional broadcast infrastructure. Our platform is being engineered as a drop-in replacement that enables:

Real-Time Alert Monitoring: Continuous polling of NOAA, NWS, and IPAWS CAP feeds with instant alert detection and processing.
Geospatial Intelligence: PostGIS-powered spatial analysis to determine which alerts affect your coverage area.
Broadcast Automation: FCC-compliant SAME encoding with automated voice synthesis for immediate on-air deployment.
Multi-Channel Distribution: Simultaneous delivery via LED signs, GPIO relays, web dashboards, and API endpoints.

Drop-In Replacement Requirements

The Master Implementation Roadmap tracks the nine capability areas that must be completed before the project can stand in for commercial hardware:

  • Reliable Baseband Capture – Unified ingest from SDR, line-in, and file sources with synchronized metering.
  • Deterministic Audio Playout – Queue-driven SAME headers, attention tones, GPIO triggers, and narration.
  • Hardware Control & Redundancy – Audited GPIO, relay, and network control with watchdog timers.
  • Security & Access Assurance – Role-based access, MFA, and tamper-evident change logs.
  • Resilient Operations – Backup, failover, and health monitoring workflows.
  • Turnkey Deployment – Setup wizards, configuration validation, and commodity hardware guidance.
  • Regulatory Compliance Support – Analytics and reporting for FCC weekly/monthly test obligations.
  • Unified Documentation – Synchronized repository and web content so operators follow the same guardrails.
  • Certification Readiness – Evidence collection, soak testing, and documentation packages that support an FCC Part 11 submission.

Reference Raspberry Pi Hardware Stack

The roadmap targets a reproducible build assembled entirely from commodity parts so integrators can swap in software for the rack-mount black boxes sold today:

Compute: Raspberry Pi 4 Model B (4 GB/8 GB) with active cooling and UPS-backed power.
Dry Contacts: Multi-relay GPIO HAT providing normally-open/closed paths for transmitter keying and tower lights.
Serial Control: RS-232 HAT or USB adapter for legacy EAS peripherals and studio automation links.
Audio I/O: Pi-compatible sound card or USB interface delivering balanced program and monitor feeds.
Confidence Monitoring: Native HDMI output for local displays plus dashboard telemetry for remote checks.
Verification: Dual SDR receivers (RTL-SDR, Airspy, etc.) validating retransmission audio and SAME headers.

Software integration, guided setup, and long-haul reliability trials tie the stack together so the project can pursue FCC certification with confidence.

System Architecture

EAS Station employs a modern, containerized architecture designed for reliability and scalability:

Dedicated polling services continuously monitor multiple CAP feed sources:

  • NOAA CAP Feeds: Real-time weather alerts from the National Weather Service
  • IPAWS Integration: Federal emergency alerts via the Integrated Public Alert and Warning System
  • Custom Sources: Support for additional CAP 1.2-compliant alert feeds
  • Polling Intelligence: Configurable intervals, retry logic, and duplicate detection
  • Parse & Validate: Full XML schema validation with detailed error reporting
  • Spatial Processing: Automatic extraction and normalization of geographic polygons, circles, and SAME/FIPS codes

PostgreSQL with PostGIS extensions provides enterprise-grade data management:

  • Geospatial Database: PostGIS 3.4+ enables advanced spatial queries and indexing
  • Alert Archival: Complete historical record with full-text search capabilities
  • Boundary Management: TIGER/Line county boundaries, weather zones, and custom service areas
  • Intersection Detection: Automatic identification of alerts affecting configured coverage areas
  • Audit Logging: Comprehensive activity tracking for compliance and debugging
  • SAME Code Registry: Built-in database of event codes, originator codes, and location codes

Data retention policies are configurable to meet FCC requirements and station policies.

FCC-compliant audio generation for on-air broadcast:

  • SAME Encoder: Pure Python implementation of EAS header/EOM burst generation
  • Attention Signal: Standard two-tone attention signal (853 Hz + 960 Hz)
  • Voice Synthesis: Optional Azure AI Speech integration for natural-sounding voice announcements
  • Audio Processing: FFmpeg and SoX for professional audio formatting (WAV, MP3, normalization)
  • Template Engine: Customizable message templates for different event types
  • Verification: Built-in audio validation and playback testing

All audio files include metadata tagging for archive management and compliance documentation.

Flexible alert delivery to multiple output channels:

  • Web Dashboard: Real-time alert visualization with interactive maps and filtering
  • LED Signage: Direct integration with Alpha protocol-compatible LED signs
  • GPIO Control: Relay triggers for transmitter control and studio automation
  • REST API: JSON endpoints for integration with external systems
  • Email Notifications: Alert digests and compliance reports via SMTP
  • SNMP Traps: Network monitoring system integration
  • SDR Verification: Optional alert verification via software-defined radio monitoring

All distribution channels include failover logic and delivery confirmation when available.

Flask-powered web application provides comprehensive system management:

  • Interactive Map: Leaflet-based map showing active alerts with polygon overlays
  • Alert History: Searchable archive with filtering by date, type, severity, and area
  • Audio Archive: Complete library of generated SAME audio with playback and download
  • Statistics Dashboard: Alert trends, system health metrics, and compliance reports
  • LED Control Panel: Real-time sign management and message scheduling
  • Admin Console: User management, system configuration, and maintenance tools
  • Compliance Tracking: FCC EAS monitoring logs and required weekly test documentation
  • System Health: Resource monitoring, database status, and service health checks

Deployment Options

EAS Station is designed for flexible deployment across various environments:

Raspberry Pi (Recommended)

Perfect for broadcast facilities with modest requirements:

  • Raspberry Pi 4 (4GB+ RAM recommended)
  • SSD storage via USB 3.0 for reliability
  • Direct GPIO access for relay control
  • Low power consumption (~10W)
  • Compact form factor for rack mounting

Dedicated Server

Ideal for large-scale deployments or multi-station networks:

  • Any Linux server (Ubuntu, Debian, RHEL)
  • 2+ CPU cores, 4GB+ RAM
  • Handles higher alert volumes
  • Better performance for audio processing
  • Supports advanced SDR verification

Cloud Hosting

  • Automated backups and disaster recovery

Release Management & Audit Trails

EAS Station treats every deployment like regulated equipment. Follow these guardrails before promoting changes:

  • Version numbering matters. Set APP_BUILD_VERSION for every build so the footer and health endpoints report the running release.
  • CHANGELOG-first workflow. Document behaviour changes under the [Unreleased] heading and call out how regressions were prevented.
  • Snapshot before upgrades. Run python tools/create_backup.py to capture the configuration, database dump, and git revision metadata.
  • Upgrade in place. Apply updates with python tools/inplace_upgrade.py to rebuild containers, run migrations, and restart services without wiping volumes.
  • Validate immediately. After the upgrade completes, spot-check ingest, SAME generation, GPIO relays, and audio playout so regressions are caught before lab exercises.

Meet the Maintainer

Timothy Kramer (KR8MER) is the creator and lead maintainer of EAS Station. Licensed in amateur radio since 2004 and upgraded to General Class in 2025, he blends public-safety field experience with modern RF engineering practices to explore resilient alerting workflows.

Kramer spent 17 years as a deputy sheriff, relying daily on mission-critical Motorola land-mobile radio systems. He now works as a full-time electrical panel electrician, applying the same attention to reliability, safety, and documentation that underpins this project.

Beyond EAS Station, Kramer is an active Skywarn spotter and maintains a lab of professional-grade radios, SDRTrunk infrastructure, ADS-B and APRS telemetry nodes, digital paging systems, Ubiquiti networking gear, drone platforms, and embedded compute projects. His goal with EAS Station is to make experimental emergency communications tooling accessible to other practitioners while reinforcing responsible, lab-only usage.
